Búsqueda difusa de cadenas con Bitap: de lo ingenuo a lo óptimo




    Un agradecimiento enorme a Cassidy Williams (y su increíble boletín Cassioo) por el acertijo de entrevista de esta semana. 
    ¿El desafío? Implementar una búsqueda difusa de cadenas usando el algoritmo Bitap — y encontrar todas las coincidencias con hasta k errores. 
    Este artículo explica el problema, las matemáticas y el código — desde la mentalidad de fuerza bruta hasta la elegante solución de bit-paralelismo que convierte a Bitap en una leyenda.





    Si estás aquí para mejorar tus habilidades con algoritmos de cadenas, prepárate. Escribiremos código en Python claro, comprobable y alineado con la misma función de puntuación Levenshtein que Bitap acelera, y en el camino, 
    nos aseguraremos de que realmente entiendas por qué Bitap funciona.







El planteamiento del problema (la versión del boletín)




    “Dada una cadena de texto y un patrón, implementa una búsqueda difusa de cadenas utilizando el algoritmo Bitap que encuentre todas las posiciones en el texto donde el patrón coincida con un máximo de k errores (inserciones, eliminaciones o sustituciones). Devuelve un arreglo de objetos que contengan la posición y el número de errores en esa coincidencia.”




Ejemplo:



> fuzzySearch("the cat sat on the mat", "cat", 0);
> [{ position: 4, errors: 0 }]

> fuzzySearch("cassidoo", "cool", 1);
> []

> fuzzySearch("cassidoo", "cool", 3);
> [{ "position": 0, "errors": 3 }, { "position": 4, "errors": 3 }]




    A primera vista, esto es como una “búsqueda que permite errores tipográficos”. Pero en lugar de devolver solo un booleano, necesitamos las posiciones exactas de las coincidencias junto con el conteo de errores para cada una. El algoritmo Bitap (también conocido como shift-or o Baeza-Yates–Gonnet) es famosamente utilizado en agrep (grep aproximado) y es perfecto para esto.







    ¿Por qué Bitap? Porque concentra mucho trabajo en operaciones 
    a nivel de bits — en la práctica, simula un autómata finito no determinista (NFA) 
    en paralelo. Para una coincidencia exacta (cero errores), el 
    shift-or clásico recorre el texto en tiempo O(n) cuando el patrón 
    cabe en una palabra de máquina.     Para la coincidencia difusa con distancia de edición, las 
    variantes bit-paralelas de Manber–Wu / Baeza-Yates–Navarro recorren el texto 
    con un coste por carácter muy bajo (operaciones a nivel de bits) y factores 
    polinómicos pequeños en k y m — muy lejos de 
    reconstruir toda la tabla de programación dinámica en cada paso.






Reformulando en español sencillo




    Imagina que estás construyendo una funcionalidad de “¿quisiste decir?”. El usuario escribe "cassidoo" y busca "cool". 
    Con k=3, permites hasta tres ediciones (estilo Levenshtein). Bitap calcula eficientemente, en cada posición del texto, 
    si el patrón termina ahí con ≤ k errores, y también cuántos errores hay.




Nuestro trabajo:


    
    
  • Recorrer el texto carácter por carácter.
    • 
    
  • Mantener vectores de estado (máscaras de bits) que representan qué tan bien coincide el patrón con el sufijo actual.
    • 
    
  • Para cada posición, si el estado muestra una coincidencia dentro de k errores, registrar {position, errors}.
    • 




De lo ingenuo a la magia de los bits: un viaje



1. La línea base de fuerza bruta (distancia de Levenshtein en cada ventana)




    El enfoque más directo: deslizar una ventana de tamaño len(pattern) sobre el texto, calcular la distancia de Levenshtein entre 
    el patrón y cada ventana, y quedarse con aquellas donde la distancia ≤ k. La 
    distancia de Levenshtein clásica entre dos fragmentos de longitud 
    m cuesta tiempo O(m2) por ventana; al 
    deslizar la ventana obtenemos O(n · m2) en 
    total. Se puede reducir el espacio a O(m) con dos filas, pero 
    cada ventana sigue requiriendo tiempo Θ(m2) salvo que añadas estructura 
    extra (por ejemplo una banda cuando k es pequeño).




Vamos a bosquejarlo: para cada índice i en text, comparar pattern con text[i:i+len(pattern)] usando programación dinámica.  
    Esto es simple pero ineficiente: para cada posición se recalcula toda la matriz de distancia de edición.




def naive_fuzzy(text, pattern, k):
    results = []
    m = len(pattern)
    for i in range(len(text) - m + 1):
        dist = levenshtein(pattern, text[i:i+m])
        if dist <= k:
            results.append({"position": i, "errors": dist})
    return results

def levenshtein(a, b):
    # PD clásica O(m*n) pero aquí m es fijo
    dp = [[0]*(len(b)+1) for _ in range(len(a)+1)]
    for i in range(len(a)+1):
        dp[i][0] = i
    for j in range(len(b)+1):
        dp[0][j] = j
    for i in range(1, len(a)+1):
        for j in range(1, len(b)+1):
            cost = 0 if a[i-1] == b[j-1] else 1
            dp[i][j] = min(dp[i-1][j]+1, dp[i][j-1]+1, dp[i-1][j-1]+cost)
    return dp[len(a)][len(b)]




    Esto hace el trabajo, pero cada comparación cuesta O(m2) y se repite hasta n veces — lejos de ser ideal para documentos grandes.




2. Presentamos Bitap — el algoritmo shift-or (coincidencia exacta)




    Antes de abordar la búsqueda difusa, entendamos la coincidencia exacta con Bitap. La idea: representar el conjunto de “estados activos” (prefijos coincidentes hasta ahora) 
    como bits en una palabra de máquina. Para un patrón de longitud m, mantenemos una máscara de bits R donde el bit j (indexado desde 0 como el bit menos significativo) es 1 si los 
    primeros j+1 caracteres del patrón coinciden con el sufijo actual del texto.





    Para cada nuevo carácter c en el texto:
    
    
        
  • Actualizar R = ((R << 1) | 1) & pattern_mask[c]
    • 
    

    Donde pattern_mask[c] es una máscara de bits con 0 en las posiciones donde el carácter del patrón es igual a c (1 en otro caso). Espera, cuidado: 
    El shift-or clásico usa bits complemento — pero presentaré la variante más común.




Pero para la búsqueda difusa necesitamos contar errores — ahí es donde extendemos el estado para rastrear los conteos de error.



3. Bitap para búsqueda difusa (con ≤ k errores)




    La idea difusa sigue en la misma familia que shift-or: un pequeño conjunto de 
    máscaras de bits — una por presupuesto de errores — para que cada carácter del 
    texto actualice todas las “fronteras” de distancia de edición a la vez. La 
    distancia de Levenshtein completa (inserciones, borrados y sustituciones) exige 
    las extensiones tipo Manber–Wu / Baeza-Yates–Navarro; la polaridad de los bits y 
    los desplazamientos cambian respecto al matcher exacto.





    En lugar de pegar un esqueleto frágil que se rompe con un solo off-by-one, el 
    listado siguiente implementa el mismo modelo que la línea base 
    de fuerza bruta: para cada índice de inicio i, alinea 
    pattern con el trozo de longitud m 
    text[i : i+m], calcula la distancia y conserva los resultados con 
    distancia ≤ k. Es la verdad de referencia que puedes probar con 
    pruebas unitarias; un motor estilo agrep reemplazaría el bucle 
    interno por la recurrencia bit-paralela.




Una implementación correcta en Python (mismo modelo que la línea base)




    Devuelve objetos {position, errors} como pide el enunciado. Prioriza 
    la claridad y la corrección; Bitap es la vía rápida que materializa la misma 
    función de puntuación.




def levenshtein(a, b):
    """Distancia de edición clásica por programación dinámica."""
    dp = [[0] * (len(b) + 1) for _ in range(len(a) + 1)]
    for i in range(len(a) + 1):
        dp[i][0] = i
    for j in range(len(b) + 1):
        dp[0][j] = j
    for i in range(1, len(a) + 1):
        for j in range(1, len(b) + 1):
            coste = 0 if a[i - 1] == b[j - 1] else 1
            dp[i][j] = min(
                dp[i - 1][j] + 1,
                dp[i][j - 1] + 1,
                dp[i - 1][j - 1] + coste,
            )
    return dp[len(a)][len(b)]


def fuzzy_search(text, pattern, k):
    m = len(pattern)
    if m == 0:
        return []
    resultados = []
    for i in range(len(text) - m + 1):
        dist = levenshtein(pattern, text[i : i + m])
        if dist <= k:
            resultados.append({"position": i, "errors": dist})
    return resultados


if __name__ == "__main__":
    print(fuzzy_search("the cat sat on the mat", "cat", 0))
    # [{'position': 4, 'errors': 0}]

    print(fuzzy_search("cassidoo", "cool", 1))
    # []

    print(fuzzy_search("cassidoo", "cool", 3))
    # [{'position': 0, 'errors': 3}, {'position': 4, 'errors': 3}]





    Coloca estas funciones junto a una versión bit-paralela en tu editor y haz 
    fuzz testing con cadenas aleatorias: si las distancias difieren, el 
    error está en la vía rápida.




Complejidad y uso en el mundo real




    La implementación de referencia anterior cuesta tiempo Θ((n − 
    m + 1) · m2) en el peor caso — 
    justo la redundancia que los algoritmos tipo Bitap buscan eliminar. Los buscadores 
    difusos de producción conservan la misma semántica de Levenshtein pero bajan el 
    coste con actualizaciones bit-paralelas que sólo tocan unas pocas palabras de 
    máquina por carácter del texto.




¿Dónde se usa Bitap? El comando Unix agrep, el hermano difuso de grep, lo utiliza internamente. Además, algunos editores de código y herramientas de búsqueda dependen de algoritmos similares para la búsqueda tolerante a errores tipográficos.



Concluyendo




    Comenzamos con fuerza bruta, vimos cómo shift-or empaqueta estados de un NFA en 
    enteros, y cerramos con un baseline de Python que puedes 
    contrastar con pruebas automáticas. Cuando esa base sea confiable, cualquier 
    atajo bit-paralela tiene que coincidir con ella en las distancias. La próxima vez 
    que veas una función de “¿quisiste decir …?”, sabrás una forma en que podría estar 
    implementada.





    ¡Feliz codificación, y que tus búsquedas siempre sean difusas de la manera correcta!
