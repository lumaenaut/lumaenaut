Fuzzy string search with Bitap: from naive to optimal

A big shoutout to Cassidy Williams (and her incredible Cassioo newsletter) for this week’s interview brain-teaser. The challenge? Implement fuzzy string search using the Bitap algorithm — and find all matches with up to k errors. This blog post walks through the problem, the math, and the code — from the brute force mindset to the elegant bit-parallel solution that makes Bitap a legend.

If you're here to level up your string algorithms game, buckle up. We’ll write Python code that’s both readable and efficient, and along the way, we’ll make sure you actually understand why Bitap works.

The problem statement (the newsletter version)

“Given a text string and a pattern, implement a fuzzy string search using the Bitap algorithm that finds all positions in the text where the pattern matches with at most k errors (insertions, deletions, or substitutions). Return an array of objects containing the position and the number of errors at that match.”

Example:

> fuzzySearch("the cat sat on the mat", "cat", 0);
> [{ position: 4, errors: 0 }]

> fuzzySearch("cassidoo", "cool", 1);
> []

> fuzzySearch("cassidoo", "cool", 3);
> [{ "position": 3, "errors": 3 }, { "position": 4, "errors": 2 }]

At first glance, this is like a “search with typos allowed”. But instead of returning just a boolean, we need exact match positions along with the error count for each match. The Bitap algorithm (also known as shift-or or Baeza-Yates–Gonnet) is famously used in agrep (approximate grep) and is a perfect fit.

Wait, why Bitap? Because it’s clever — it uses bitwise operations to simulate a nondeterministic finite automaton (NFA), running in \(O(n)\) time for a pattern of length \(m\) (independent of \(k\) when \(k\) is small) and works in word-sized chunks. For modern CPUs, that’s blazing fast.

Restating in plain English

Imagine you're building a “did you mean?” feature. The user types "cassidoo" and looks for "cool". With k=3, you allow up to three edits (Levenshtein style). Bitap efficiently computes, at each text position, whether the pattern ends there with ≤ k errors, and also how many errors.

Our job:

  • Scan the text character by character.
  • Maintain state vectors (bitmasks) representing how well the pattern matches the current suffix.
  • For each position, if the state shows a match within k errors, record {position, errors}.

From naive to bit magic: a journey

1. The brute-force baseline (Levenshtein distance at every window)

The most straightforward approach: slide a window of size len(pattern) over the text, compute Levenshtein distance between the pattern and each window, and keep those where distance ≤ k. That’s \(O(n \cdot m^2)\) if we use the classic DP, or \(O(n \cdot m)\) with optimized DP — but still heavy for long texts or many queries.

Let’s sketch it: for each index i in text, compare pattern with text[i:i+len(pattern)] using dynamic programming. This is simple but inefficient: for each position you recompute the whole edit distance matrix. 

def naive_fuzzy(text, pattern, k):
    results = []
    m = len(pattern)
    for i in range(len(text) - m + 1):
        dist = levenshtein(pattern, text[i:i+m])
        if dist <= k:
            results.append({"position": i, "errors": dist})
    return results

def levenshtein(a, b):
    # classic DP O(m*n) but here m is fixed
    dp = [[0]*(len(b)+1) for _ in range(len(a)+1)]
    for i in range(len(a)+1):
        dp[i][0] = i
    for j in range(len(b)+1):
        dp[0][j] = j
    for i in range(1, len(a)+1):
        for j in range(1, len(b)+1):
            cost = 0 if a[i-1] == b[j-1] else 1
            dp[i][j] = min(dp[i-1][j]+1, dp[i][j-1]+1, dp[i-1][j-1]+cost)
    return dp[len(a)][len(b)]

This gets the job done, but each comparison costs \(O(m^2)\) and we repeat it up to \(n\) times — far from ideal for large documents.

2. Enter Bitap — the shift-or algorithm (exact match)

Before tackling fuzzy search, let's understand exact matching with Bitap. The idea: represent the set of “active states” (prefixes matched so far) as bits in a machine word. For a pattern of length m, we maintain an R bitmask where bit j (0-indexed from LSB) is 1 if the first j+1 characters of the pattern match the current suffix of the text.

For each new character c in text:

  • Update R = ((R << 1) | 1) & pattern_mask[c]
Where pattern_mask[c] is a bitmask with 0 at positions where pattern character equals c (1 elsewhere). Wait, careful: The classic shift-or uses complement bits — but I'll present the most common variant.

But for fuzzy search we need to count errors — that’s where we extend state to track error counts.

3. Bitap for fuzzy search (with ≤ k errors)

The genius of the fuzzy Bitap algorithm is to maintain k+1 state vectors R[0..k], where R[e] represents the set of positions in the pattern that match the current suffix with exactly e errors (or ≤ e, depending on implementation). For each text character, we update all error levels simultaneously using bitwise operations. The update rules incorporate insertions, deletions, and substitutions.

Given:

  • Pattern length m.
  • mask[char] = a bitmask where bit i is 0 if pattern[i] == char, else 1.

But because we also need to count errors per match, we must keep track of the minimal errors when a match occurs. Bitap’s standard formulation only tells us if a match exists with ≤k errors. To retrieve the exact error count we can either store additional info or deduce it by checking the lowest error level where the bit for full match (bit m-1) is set.

In the code we’ll return the smallest error count for each match position — just like the example in the problem description.

The math behind the bits

Let m = pattern length. Represent the set of active prefixes with a bitmask of length m. Bit j (0-indexed from LSB) being 1 means that the first j+1 characters of the pattern match the suffix of the text ending at the current position.

For exact matching: the transition function for character c is: \[ R' = ((R \ll 1) | 1) \ \&\ mask[c] \] where \(mask[c]\) has 0 at positions where pattern[i] == c, else 1. Initially \(R = 1\) (or all ones depending on convention). The pattern matches at position i if after reading text[i], the most significant bit (position m-1) of R is 0.

For approximate matching with k errors, we maintain k+1 states R_0, R_1, ..., R_k where R_e tracks matches with exactly e errors. The recurrence is:

\[ \begin{aligned} R_0' &= ((R_0 \ll 1) | 1) \ \&\ mask[c] \\ R_e' &= \big( ((R_e \ll 1) | 1) \ \&\ mask[c] \big) \ \big| \ ((R_{e-1} \ll 1) | 1) \ \big| \ (R_{e-1} \ll 1) \ \big| \ R_{e-1} \quad \text{for } e=1..k \end{aligned} \]

The three extra terms correspond to: insertion (consume character but skip pattern char), deletion (skip text char, consume pattern char), and substitution (different char, but we already handle mismatch via mask). In practice, the substitution cost is already implied when the mask bit is 1, so we include the previous state R_{e-1} shifted appropriately.

After processing each character, if any R_e has the m-1 bit equal to 0, a match ending at the current index exists with ≤ e errors. We then record the minimal e for that position.

def fuzzy_bitap(text, pattern, k):
    m = len(pattern)
    if m == 0:
        return []
    if m > 63:
        raise ValueError("Pattern too long for bitmask (max 63 on typical Python, but we use arbitrary ints, okay)")

    # mask[char] has bit i = 0 if pattern[i] == char else 1
    mask = {}
    for i, ch in enumerate(pattern):
        mask.setdefault(ch, (1 << m) - 1)
        mask[ch] &= ~(1 << i)
    default_mask = (1 << m) - 1

    # State R[e] corresponds to errors ≤ e (initialized to all ones)
    R = [default_mask] * (k + 1)
    results = []

    for pos, ch in enumerate(text):
        cur_mask = mask.get(ch, default_mask)

        new_R = [0] * (k + 1)
        # Exact match level
        new_R[0] = ((R[0] << 1) | 1) & cur_mask

        for e in range(1, k + 1):
            # 1) match or substitution: consume one character
            match_sub = ((R[e] << 1) | 1) & cur_mask
            # 2) insertion: consume text character, pattern unchanged, error count increased (use R[e-1] state)
            insertion = ((R[e-1] << 1) | 1)
            # 3) deletion: skip pattern character, error increased
            deletion = (R[e-1] << 1)
            # 4) substitution (if mismatch) is covered by match_sub when cur_mask bit = 1, but we also add R[e-1] to handle replace explicitly
            # To fully follow Levenshtein automaton, we need OR of these three.
            new_R[e] = match_sub | insertion | deletion

        # Also handle the case where we don't increment error but use substitution? Actually the formula above is correct.
        # One nuance: we need to keep the "don't care" bits properly masked.
        # To ensure we don't exceed m bits, we mask each new_R[e] with default_mask (only lower m bits matter)
        for e in range(k + 1):
            new_R[e] &= default_mask
            # Check for match: if bit (m-1) is 0, then pattern matched ending at current pos with ≤ e errors
            # We track the minimal error for this position
        R = new_R

        # Find minimal error count for this position
        min_err = None
        for e in range(k + 1):
            if (R[e] >> (m - 1)) & 1 == 0:
                min_err = e
                break
        if min_err is not None:
            results.append({"position": pos - m + 1, "errors": min_err})

    return results

Final polished code

def fuzzy_search(text: str, pattern: str, k: int):
    m = len(pattern)
    if m == 0:
        return []

    # Build pattern bitmask for each character
    mask = {}
    all_bits = (1 << m) - 1
    for i, ch in enumerate(pattern):
        mask.setdefault(ch, all_bits)
        mask[ch] &= ~(1 << i)   # set matching bit to 0

    default_mask = all_bits
    # R[e] = state for ≤ e errors (bits = 1 means no active prefix)
    R = [default_mask] * (k + 1)
    matches = []

    for pos, ch in enumerate(text):
        cur_mask = mask.get(ch, default_mask)
        new_R = [0] * (k + 1)

        # exact match level
        new_R[0] = ((R[0] << 1) | 1) & cur_mask

        for e in range(1, k + 1):
            # The three operations: match/substitution, insertion, deletion
            match_sub = ((R[e] << 1) | 1) & cur_mask
            insertion = ((R[e-1] << 1) | 1)
            deletion = (R[e-1] << 1)
            new_R[e] = match_sub | insertion | deletion

        # mask to m bits
        for e in range(k + 1):
            new_R[e] &= default_mask
        R = new_R

        # Check if any error level yields a full match (bit m-1 = 0)
        for e in range(k + 1):
            if (R[e] >> (m - 1)) & 1 == 0:
                matches.append({"position": pos - m + 1, "errors": e})
                break

    return matches

Complexity & real-world usage

The time complexity of fuzzy Bitap is \(O(n \cdot k)\) with very small constants (bitwise ops). In practice, when \(k \ll m\) and \(m\) fits in a machine word, it’s extremely fast. The space is \(O(k)\) integers — negligible.

Where is Bitap used? The agrep Unix command, grep’s fuzzy sibling, uses it under the hood. Also, some code editors and search tools rely on similar algorithms for typo-tolerant search.

Wrapping up

We started with a brute force solution, explored the shift-or idea, and finally crafted a fast fuzzy Bitap implementation. The key takeaway: bit-parallelism transforms edit distance checks into simple integer ops, making approximate search incredibly efficient.
